# SES Email OTP Service

A production-grade **email one-time-password (verification code / 邮箱验证码) service** built
on **AWS SES, API Gateway (HTTP API), Lambda, DynamoDB and Secrets Manager**, defined entirely
in **AWS CDK (TypeScript)**.

A client asks for a code → the backend generates one, stores only its salted HMAC, and emails
it via SES → the user receives the code and submits it back → the backend verifies it and
issues a short-lived signed verification token.

A browser UI is also served at `/` (a Web Lambda), and **CloudFront fronts everything** as a
single origin so the UI and API are same-origin (no CORS).

```
Browser ──▶ CloudFront ──▶ API Gateway (HTTP API, throttled + access logs)
                                │
          ┌─────────────────────┼─────────────────────┐
   GET /              POST /v1/otp/request      POST /v1/otp/verify
      │                     │                          │
  Lambda Web         Lambda RequestOtp          Lambda VerifyOtp   (Node 20 · TS · esbuild · Powertools · X-Ray)
                           │                          │
                           ├── DynamoDB (otp-codes, TTL auto-expiry, encrypted)
                           ├── SES v2 SendEmail (verified sender → recipient)
                           └── Secrets Manager (HMAC pepper — generated by CDK, fetched at runtime)
```

After `cdk deploy`, open the **`SiteUrl`** output (the CloudFront URL) in a browser to use the
demo UI, or call the API directly at the **`ApiBaseUrl`** output.

## Security model

| Concern | Control |
| --- | --- |
| Code at rest | Never stored in clear. Stored as `HMAC-SHA256(salt:purpose:code)` keyed by a server-side **pepper** in Secrets Manager. A DB leak alone does not reveal codes. |
| Pepper handling | Generated by CDK into Secrets Manager; fetched + cached at runtime. **Never** committed, logged, or exposed as a plaintext env var — only its ARN is passed to the Lambdas. |
| Timing attacks | Constant-time comparison (`crypto.timingSafeEqual`). |
| Replay / reuse | Codes are **single-use** (atomic `consumed` flag via DynamoDB conditional write). |
| Brute force | Max **5** verify attempts per code, then the code is locked. |
| Abuse / spam | Resend cooldown **60s**; max **5** sends per email per hour (tracked in DynamoDB); API Gateway stage throttling as an outer guard. |
| Expiry | Codes expire after **10 min** (DynamoDB TTL auto-deletes the record). |
| Enumeration | Uniform responses; raw IPs never stored (only a SHA-256 hash). |
| Input | Strict validation with `zod`. |
| IAM | Least privilege — `request` may send mail only as the configured sender; `verify` has no SES access at all. |
| Secrets in git | Enforced by `.gitignore` + config-injection design. No credentials or secrets are committed. |

## API

### `POST /v1/otp/request`
Request body:
```json
{ "email": "user@example.com", "purpose": "login" }
```
| Status | Meaning |
| --- | --- |
| `202` | `{ "status": "sent", "expiresInSeconds": 600, "requestId": "..." }` |
| `400` | Invalid email |
| `429` | Rate-limited (`retryAfterSeconds`, plus `Retry-After` header) |

### `POST /v1/otp/verify`
Request body:
```json
{ "email": "user@example.com", "code": "123456" }
```
| Status | Meaning |
| --- | --- |
| `200` | `{ "status": "verified", "verificationToken": "...", "expiresInSeconds": 300 }` |
| `401` | `{ "status": "invalid", "remainingAttempts": n }` |
| `410` | Code expired / not found / already used |
| `429` | Too many attempts — code locked |

The `verificationToken` is a stateless HMAC token (signed with the same pepper) a downstream
service can trust as proof this email was verified within the last 5 minutes.

## Configuration

Nothing sensitive lives in source. Configuration is supplied at deploy time:

- **Sender email** — passed via CDK context `-c senderEmail=...` (non-secret operational config).
- **HMAC pepper** — generated by CDK directly into Secrets Manager; never seen by a human.

Tunables (TTL, code length, attempt/rate limits) are Lambda env vars set in
`lib/otp-service-stack.ts` and validated by `src/lib/config.ts`.

## Deploy

> Deploys are pinned to `--profile temp-account` so they only ever target the authorized
> account. They will **not** touch your local default AWS account.

Set your own addresses as shell variables (do not hard-code them):

```bash
export SENDER_EMAIL="you@your-verified-domain.com"     # a verified SES sender
export TEST_EMAIL="recipient@example.com"              # a mailbox you can read

# 1. Verify the SES identities (sandbox requires BOTH sender and recipient).
./scripts/verify-identity.sh "$SENDER_EMAIL"           # click the link AWS emails
./scripts/verify-identity.sh "$TEST_EMAIL"             # click the link

# 2. Deploy.
./scripts/deploy.sh "$SENDER_EMAIL"
#   → outputs SiteUrl, ApiBaseUrl, TableName, PepperSecretArn
```

## SES sandbox & production access

This account's SES is in the **sandbox**:
- You can only send **to** and **from** verified identities.
- Quota: 200 emails/day, 1 message/sec.

To run for real users, request production access (removes the verified-recipient restriction
and raises quota) — AWS Console → SES → *Account dashboard* → *Request production access*, or:
```bash
aws sesv2 put-account-details \
  --production-access-enabled \
  --mail-type TRANSACTIONAL \
  --website-url https://your-site.example \
  --use-case-description "Transactional email verification codes for sign-in/sign-up." \
  --profile temp-account --region us-east-1
```

## Deliverability (don't land in spam)

Sending succeeds (SES `Delivery` event fires) but the mail can still land in the recipient's
**spam folder**. This is an email-authentication problem, not a code bug:

- With an **email-address** identity (e.g. a personal mailbox at a public provider), the message
  is signed by `amazonses.com`, so SPF/DKIM do **not align** with the From domain. The provider
  accepts it but files it as spam. Sending *as* a large corporate domain is worse — such domains
  often publish DMARC `p=quarantine`, so unaligned mail is quarantined outright.
- **Fix (production):** authenticate a **domain identity you control** and enable DKIM:
  ```bash
  aws sesv2 create-email-identity --email-identity yourdomain.com \
    --profile temp-account --region us-east-1
  # → add the 3 CNAME records it returns at your DNS provider; SES signs with aligned DKIM.
  ```
  Then send `From: no-reply@yourdomain.com`. Optionally add a DMARC record and a custom
  MAIL FROM domain for SPF alignment. With aligned DKIM, mail reaches the inbox.

To diagnose where a message actually went, attach a configuration set with a CloudWatch event
destination and inspect the `Send`/`Delivery`/`Bounce`/`Reject` metrics — `Delivery` with no
`Bounce` means the provider accepted it (and any "missing" mail is in spam/promotions).

## Secret / PII guard

`scripts/pii-guard.sh` blocks real emails, AWS account IDs, access keys, and deployed
hostnames/ARNs from entering the repo. It runs in CI and can be installed as a pre-commit hook:

```bash
ln -sf ../../scripts/pii-guard.sh .git/hooks/pre-commit
```

Generic placeholders (`example.com`, `<id>`) are allowed; real values are injected at
deploy/runtime, never committed.

## CI/CD

`.github/workflows/ci.yml` runs on every push/PR: PII guard → install → type-check → unit
tests → `cdk synth`. **No AWS credentials are required** — synth renders CloudFormation locally.

For automated deployment, the recommended production path is **GitHub Actions OIDC** assuming a
deploy role (no long-lived keys in CI). In this workshop account that role may be restricted by
SCP, so deployment is performed locally with `--profile temp-account` via `scripts/deploy.sh`.

## Tear down

```bash
./scripts/destroy.sh "$SENDER_EMAIL"
```

> The DynamoDB table and the pepper secret use `RemovalPolicy.DESTROY` so the temp account
> cleans up fully. For real production, switch these to `RETAIN`.
